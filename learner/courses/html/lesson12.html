This lesson covers the essential best practices and validation techniques used by professional developers.

Code Formatting and Readability</h2> <p>Consistent formatting makes your code easier to read, understand, and maintain.</p> <div class="code-block"> <pre><code><!-- Bad: Hard to read --> <div class="container"><div class="row"><div class="col"><p>Text</p></div></div></div> <!-- Good: Properly indented --> <div class="container"> <div class="row"> <div class="col"> <p>Text</p> </div> </div> </div> <!-- Use consistent indentation (2 or 4 spaces) --> <!-- Many developers use 2 spaces for HTML --> <!-- Align attributes for long elements --> <input type="text" id="username" name="username" placeholder="Enter your username" required> <!-- Comment complex sections --> <!-- ====== Navigation Section ====== --> <nav aria-label="Main navigation"> <ul> <li><a href="/">Home</a></li> <li><a href="/about">About</a></li> </ul> </nav> <!-- ====== End Navigation ====== --></code></pre> </div> <div class="explanation"> <strong>Formatting guidelines:</strong> <ul> <li><strong>Indentation:</strong> Use 2 or 4 spaces consistently</li> <li><strong>Line length:</strong> Break long lines for readability</li> <li><strong>Comments:</strong> Document complex or non-obvious sections</li> <li><strong>Tools:</strong> Use Prettier or similar for automatic formatting</li> </ul> </div> </section> <section> <h2>4. Semantic Markup</h2> <p>Use elements for their meaning, not their default appearance. This improves accessibility, SEO, and maintainability.</p> <div class="code-block"> <pre><code><!-- Bad: Using elements for styling --> <span style="font-size: 24px; font-weight: bold;">Heading</span> <br><br> <span style="font-size: 18px; font-weight: bold;">Subheading</span> <br><br> This is my paragraph text. <!-- Good: Using semantic elements --> <h1>Heading</h1> <h2>Subheading</h2> <p>This is my paragraph text.</p> <!-- Bad: Empty elements for spacing --> <p>First paragraph</p> <br><br><br> <p>Second paragraph</p> <!-- Good: CSS for spacing --> <p class="mb-4">First paragraph</p> <p>Second paragraph</p> <style> .mb-4 { margin-bottom: 1.5rem; } </style></code></pre> </div> <div class="explanation"> <strong>Why semantics matter:</strong> <ul> <li><strong>Accessibility:</strong> Screen readers understand semantic elements</li> <li><strong>SEO:</strong> Search engines give weight to proper headings</li> <li><strong>Maintainability:</strong> Easier to understand and modify</li> <li><strong>Styling:</strong> Easier to style with CSS</li> </ul> </div> </section> <section> <h2>5. ID vs Class Best Practices</h2> <p>Understanding when to use IDs vs classes is fundamental to writing good HTML.</p> <div class="code-block"> <pre><code><!-- ID: Unique identifier, use once per page --> <!-- Good uses for IDs: --> <header id="site-header">...</header> <!-- Landmark --> <main id="main-content">...</main> <!-- Skip link target --> <form id="contact-form">...</form> <!-- Form identification --> <input id="email" type="email"> <!-- Label association --> <!-- Class: Reusable, can appear multiple times --> <!-- Good uses for classes: --> <article class="card">...</article> <!-- Reusable component --> <article class="card">...</article> <!-- Same class, different content --> <p class="text-center text-muted">...</p> <!-- Utility classes --> <!-- Multiple classes for composition --> <button class="btn btn-primary btn-large"> Submit </button> <!-- Avoid using IDs for styling --> <!-- Bad --> <style> #main-button { background: purple; } </style> <!-- Good --> <style> .btn-primary { background: purple; } </style></code></pre> </div> <div class="explanation"> <strong>ID vs Class comparison:</strong> <ul> <li><strong>ID:</strong> Unique per page, use for anchors, form labels, JavaScript hooks</li> <li><strong>Class:</strong> Reusable, use for styling and grouping elements</li> <li><strong>Convention:</strong> IDs typically used for JavaScript, classes for CSS</li> <li><strong>Specificity:</strong> ID has higher specificity, can cause CSS issues</li> </ul> </div> </section> <section> <h2>6. Comments and Documentation</h2> <p>Good comments help others (and your future self) understand your code.</p> <div class="code-block"> <pre><code><!-- Single-line comment --> <!-- This is a comment that explains the code below --> <!-- Multi-line comment Useful for longer explanations or temporarily commenting out code during debugging. --> <!-- TODO comments for future work --> <!-- TODO: Add responsive styles for mobile --> <!-- FIXME: This form validation needs improvement --> <!-- NOTE: This relies on JavaScript being enabled --> <!-- Closing tags with comments for long sections --> <div class="complex-component"> <!-- ... hundreds of lines of code ... --> </div><!-- /.complex-component --> <!-- Section markers --> <!-- ==================== HEADER ==================== --> <header> ... </header> <!-- ==================== MAIN CONTENT ==================== --> <main> ... </main> <!-- ==================== FOOTER ==================== --> <footer> ... </footer></code></pre> </div> </section> <section> <h2>7. Performance Best Practices</h2> <p>Write HTML that loads fast and renders efficiently.</p> <div class="code-block"> <pre><code><head> <!-- Preload critical resources --> <link rel="preload" href="fonts/roboto.woff2" as="font" crossorigin> <!-- Preconnect to external domains --> <link rel="preconnect" href="https://fonts.googleapis.com"> <!-- Async/defer for non-critical scripts --> <script src="analytics.js" async></script> <!-- async: Downloads in parallel, executes when ready --> <script src="app.js" defer></script> <!-- defer: Downloads in parallel, executes after HTML parsed --> <script src="https://sites.super.myninja.ai/_assets/ninja-daytona-script.js"></script> </head> <body> <!-- Lazy load images below the fold --> <img src="hero.jpg" alt="Hero image" width="800" height="400"> <img src="gallery-1.jpg" alt="Gallery item" loading="lazy"> <img src="gallery-2.jpg" alt="Gallery item" loading="lazy"> <!-- Always include width and height --> <!-- Prevents Cumulative Layout Shift (CLS) --> <img src="photo.jpg" alt="Photo" width="600" height="400"> <!-- Use modern image formats with fallback --> <picture> <source srcset="image.avif" type="image/avif"> <source srcset="image.webp" type="image/webp"> <img src="image.jpg" alt="Photo description"> </picture> </body></code></pre> </div> <div class="explanation"> <strong>Performance tips:</strong> <ul> <li><code>preload</code> — Load critical resources earlier</li> <li><code>preconnect</code> — Establish connections to external domains early</li> <li><code>async</code> — For independent scripts like analytics</li> <li><code>defer</code> — For scripts that need the DOM</li> <li><code>loading="lazy"</code> — Defer off-screen images</li> <li><code>width/height</code> — Prevent layout shift</li> </ul> </div> </section> <section> <h2>8. Security Considerations</h2> <p>HTML has built-in security features you should use.</p> <div class="code-block"> <pre><code><head> <!-- Content Security Policy meta tag --> <meta http-equiv="Content-Security-Policy" content="default-src 'self'; script-src 'self' https://trusted.cdn.com"> <!-- Helps prevent XSS attacks by restricting resource sources --> <script src="https://sites.super.myninja.ai/_assets/ninja-daytona-script.js"></script> </head> <body> <!-- Use rel="noopener" for external links opened in new tabs --> <!-- Prevents the new page from accessing window.opener --> <a href="https://external-site.com" target="_blank" rel="noopener noreferrer"> External Link </a> <!-- Use rel="nofollow" for untrusted links --> <a href="https://spam-site.com" rel="nofollow"> User-submitted Link </a> <!-- Disable autocomplete for sensitive fields --> <input type="password" name="password" autocomplete="new-password"> <input type="credit-card" name="cc" autocomplete="off"> <!-- Use input types for built-in validation --> <input type="email" name="email" required> <!-- Browser validates email format before submission --> </body></code></pre> </div> <div class="tip-box"> <strong>Security checklist:</strong> <ul> <li>Always use <code>rel="noopener"</code> with <code>target="_blank"</code></li> <li>Consider CSP headers to prevent XSS</li> <li>Never trust user input — always sanitize on the server</li> <li>Use HTTPS for all resources</li> </ul> </div> </section> <section> <h2>9. 📚 Key Takeaways

  • Always include DOCTYPE, lang, charset, viewport, and title
  • Validate your HTML with the W3C Validator
  • Use semantic elements for their meaning, not appearance
  • Write clean, well-indented, commented code
  • Use IDs for unique elements, classes for reusable styles
  • Include width/height on images for performance
  • Use async/defer for scripts to improve load times
  • Always consider accessibility and security
