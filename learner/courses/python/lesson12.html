Lesson 12: Error Handling

Handle exceptions gracefully with try, except, and custom exceptions

Understanding Exceptions

Exceptions are errors that occur during program execution. Python uses exceptions to signal that something unexpected happened. Proper error handling makes your programs robust and user-friendly by gracefully handling problems instead of crashing.

Try-Except Basics

The try-except block lets you catch and handle exceptions. Code that might raise an exception goes in the try block, and handling code goes in except blocks.

Basic Exception Handling:

# Basic try-except
try:
    number = int(input("Enter a number: "))
    print(f"You entered: {number}")
except ValueError:
    print("That's not a valid number!")

# Catching multiple exceptions
try:
    value = int(input("Enter a number: "))
    result = 10 / value
    print(f"Result: {result}")
except ValueError:
    print("Invalid number!")
except ZeroDivisionError:
    print("Cannot divide by zero!")

# Catching multiple exceptions together
try:
    file = open("nonexistent.txt", "r")
    content = file.read()
except (FileNotFoundError, PermissionError) as e:
    print(f"Error: {e}")

# Catch all exceptions (use sparingly!)
try:
    risky_operation()
except Exception as e:
    print(f"An error occurred: {e}")
    # Better to catch specific exceptions

# The 'as' keyword captures the exception object
try:
    result = 10 / 0
except ZeroDivisionError as error:
    print(f"Error: {error}")
    print(f"Error type: {type(error).__name__}")

Lines 2-6: The try block contains code that might fail. Except catches specific exceptions.

Lines 9-16: Multiple except blocks handle different types of exceptions.

Lines 19-22: Group related exceptions in a tuple to handle them the same way.

Lines 25-28: Catching Exception catches everything but should be used sparingly.

Lines 31-35: The as keyword captures the exception object for inspection.

The Full Try-Except Structure

Python's try statement can include else and finally blocks for complete control over execution flow.

Complete Try Statement:

def divide_numbers(a, b):
    try:
        result = a / b
    except ZeroDivisionError:
        print("Cannot divide by zero!")
        return None
    except TypeError:
        print("Both arguments must be numbers!")
        return None
    else:
        # Runs if no exception occurred
        print("Division successful!")
        return result
    finally:
        # Always runs, regardless of exceptions
        print("Operation complete.")

# Test cases
print(divide_numbers(10, 2))
# Division successful.
# Operation complete.
# 5.0

print(divide_numbers(10, 0))
# Cannot divide by zero!
# Operation complete.
# None

# Practical example with file handling
def read_config(filename):
    config = {}
    try:
        with open(filename) as f:
            config = parse_config(f)
    except FileNotFoundError:
        print(f"Config file not found: {filename}")
    except PermissionError:
        print(f"Permission denied: {filename}")
    else:
        print("Config loaded successfully")
    finally:
        print("Config loading complete")
    return config

# While-else and for-else with exceptions
def find_item(items, target):
    for item in items:
        if item == target:
            return item
    else:
        # Runs if loop completed without finding
        raise ValueError(f"{target} not found in items")

Lines 4-12: The else block runs only if no exception occurred.

Lines 13-15: The finally block always runs, used for cleanup.

Lines 22-28: Even with early return, finally still executes.

Lines 31-42: Real-world pattern for file handling with multiple exception types.

Lines 45-52: The else clause in loops runs if no break occurred.

💡 When to Use Each Block

Use try for risky code, except for handling errors, else for code that runs on success, and finally for cleanup that must always happen (closing files, releasing resources).

Raising Exceptions

You can raise exceptions using the raise keyword. This is useful for validating inputs or signaling error conditions in your code.

Raising Exceptions:

# Raise a simple exception
def set_age(age):
    if age < 0:
        raise ValueError("Age cannot be negative")
    return age

try:
    set_age(-5)
except ValueError as e:
    print(e)  # Age cannot be negative

# Re-raising exceptions
def process_data(data):
    try:
        result = parse(data)
    except ValueError:
        print("Logging error...")
        raise  # Re-raise the same exception

# Raise with custom message
def withdraw(balance, amount):
    if amount > balance:
        raise ValueError(
            f"Insufficient funds: {balance} available, "
            f"{amount} requested"
        )
    return balance - amount

# Raise different exception types
def validate_user(user):
    if not user:
        raise TypeError("User cannot be None")
    if not user.get("name"):
        raise ValueError("User must have a name")
    if user.get("age", 0) < 0:
        raise ValueError("Age cannot be negative")

# Chaining exceptions (Python 3+)
try:
    try:
        value = int("not a number")
    except ValueError as e:
        raise RuntimeError("Processing failed") from e

# Exception chaining shows the original cause
# RuntimeError: Processing failed
# The above exception was the direct cause of...

Lines 3-8: Use raise to signal error conditions in your functions.

Lines 11-15: Use bare raise to re-raise the current exception after logging.

Lines 18-24: Provide descriptive error messages with context.

Lines 27-33: Raise different exception types for different error conditions.

Lines 36-42: Use raise ... from e to chain exceptions and preserve context.

Custom Exceptions

Create your own exception classes for domain-specific errors. Custom exceptions make your code more expressive and easier to handle.

Creating Custom Exceptions:

# Basic custom exception
class InsufficientFundsError(Exception):
    """Raised when account has insufficient funds."""
    pass

# Custom exception with attributes
class BankError(Exception):
    """Base exception for bank operations."""
    def __init__(self, message, code=None):
        self.message = message
        self.code = code
        super().__init__(self.message)

class InsufficientFundsError(BankError):
    """Raised when withdrawal exceeds balance."""
    def __init__(self, balance, amount):
        self.balance = balance
        self.amount = amount
        super().__init__(
            f"Insufficient funds: {balance} available, "
            f"{amount} requested",
            code="INSUFFICIENT_FUNDS"
        )

# Using custom exceptions
class BankAccount:
    def __init__(self, balance=0):
        self.balance = balance
    
    def withdraw(self, amount):
        if amount > self.balance:
            raise InsufficientFundsError(self.balance, amount)
        self.balance -= amount
        return amount

# Handling custom exceptions
try:
    account = BankAccount(100)
    account.withdraw(150)
except InsufficientFundsError as e:
    print(f"Error: {e}")
    print(f"Balance: {e.balance}")
    print(f"Code: {e.code}")

# Exception hierarchy
class AppError(Exception):
    """Base exception for the application."""

class DatabaseError(AppError):
    """Database-related errors."""

class ConnectionError(DatabaseError):
    """Connection failures."""

class QueryError(DatabaseError):
    """Query execution failures."""

# Catch by hierarchy
try:
    raise ConnectionError("Failed to connect")
except DatabaseError:  # Catches any DatabaseError subclass
    print("Database operation failed")

Lines 3-5: Simple custom exception inheriting from Exception.

Lines 8-12: Custom exception with additional attributes.

Lines 15-24: Specialized exception with contextual attributes.

Lines 27-34: Use custom exceptions in your classes.

Lines 37-41: Access custom attributes when handling the exception.

Lines 44-57: Create a hierarchy of exceptions for organized error handling.

Common Built-in Exceptions

Python has many built-in exceptions. Understanding them helps you write better error handling code.

Common Exceptions:

# ValueError - right type, inappropriate value
int("abc")  # ValueError

# TypeError - wrong type for operation
"hello" + 5  # TypeError

# IndexError - list index out of range
[1, 2, 3][10]  # IndexError

# KeyError - dictionary key not found
{"a": 1}["b"]  # KeyError

# FileNotFoundError - file doesn't exist
open("missing.txt")  # FileNotFoundError

# ZeroDivisionError - division by zero
10 / 0  # ZeroDivisionError

# AttributeError - object lacks attribute
"hello".unknown_method()  # AttributeError

# NameError - variable not defined
print(undefined_var)  # NameError

# Handle specific exceptions appropriately
def safe_get(dictionary, key, default=None):
    try:
        return dictionary[key]
    except KeyError:
        return default

def safe_divide(a, b):
    try:
        return a / b
    except ZeroDivisionError:
        return float('inf')

def safe_index(lst, index):
    try:
        return lst[index]
    except IndexError:
        return None

Lines 5-6: TypeError occurs when types are incompatible.

Lines 8-9: IndexError occurs when accessing non-existent list index.

Lines 11-12: KeyError occurs when dictionary key doesn't exist.

Context Managers

Context managers handle setup and cleanup automatically using the with statement. They're perfect for resource management like files, connections, and locks.

Creating Context Managers:

# Using context manager with files
with open("file.txt", "w") as f:
    f.write("Hello")
# File automatically closed

# Creating context manager with class
class Timer:
    def __enter__(self):
        import time
        self.start = time.time()
        return self
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        self.end = time.time()
        self.elapsed = self.end - self.start
        print(f"Elapsed time: {self.elapsed:.2f}s")
        return False  # Don't suppress exceptions

# Using the custom context manager
with Timer() as t:
    sum(range(1000000))
# Elapsed time: 0.05s

# Creating context manager with decorator
from contextlib import contextmanager

@contextmanager
def open_file(filename, mode="r"):
    f = open(filename, mode)
    try:
        yield f
    finally:
        f.close()

with open_file("file.txt") as f:
    content = f.read()

# Database connection context manager
@contextmanager
def database_connection(host):
    conn = connect_to_db(host)
    try:
        yield conn
    finally:
        conn.close()

# Suppress specific exceptions
from contextlib import suppress

with suppress(FileNotFoundError):
    os.remove("file.txt")  # Won't raise if file doesn't exist

Lines 2-4: Files are the most common use of context managers.

Lines 7-17: Class-based context manager with __enter__ and __exit__.

Lines 15-16: __exit__ receives exception info; return True to suppress.

Lines 24-32: Use @contextmanager decorator for simpler creation.

Line 28: yield provides the value to the as variable.

Lines 43-45: suppress context manager ignores specific exceptions.

Best Practices

Follow these best practices for effective error handling.

Error Handling Guidelines:

# 1. Be specific about exceptions
# Bad
try:
    do_something()
except:  # Catches everything, including KeyboardInterrupt
    pass

# Good
try:
    do_something()
except (ValueError, TypeError) as e:
    handle_error(e)

# 2. Don't use exceptions for flow control
# Bad
try:
    item = my_list[index]
except IndexError:
    item = None

# Good
if index < len(my_list):
    item = my_list[index]
else:
    item = None

# 3. Clean up resources properly
# Good - use context managers
with open("file.txt") as f:
    process(f)

# 4. Provide helpful error messages
# Bad
if value < 0:
    raise ValueError("Invalid value")

# Good
if value < 0:
    raise ValueError(
        f"Value must be non-negative, got {value}"
    )

# 5. Log exceptions before re-raising
import logging
logger = logging.getLogger(__name__)

def process(data):
    try:
        return transform(data)
    except ValueError as e:
        logger.error(f"Processing failed: {e}")
        raise

# 6. Use custom exceptions for domain errors
class ValidationError(Exception):
    """Input validation failed."""
    pass

Lines 3-10: Never use bare except: - it catches system exceptions too.

Lines 13-22: Use conditional checks instead of exceptions for expected conditions.

Lines 25-27: Always clean up resources with context managers.

Lines 30-38: Include relevant information in error messages.

Lines 41-48: Log exceptions for debugging before re-raising.
